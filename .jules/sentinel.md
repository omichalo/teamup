## 2025-05-15 - Hardened CSRF Protection with HMAC-SHA256 and Timing-Safe Validation
**Vulnerability:** The previous CSRF implementation used simple base64 concatenation of user UID and secret, which was susceptible to secret leakage and lacked cryptographic integrity. Furthermore, it relied on simple string comparison, which is vulnerable to timing attacks. Several administrative POST endpoints also lacked any CSRF origin validation.
**Learning:** Even with a secret, simple concatenation does not provide strong integrity. Using a signed Double-Submit Cookie pattern with HMAC-SHA256 provides a much stronger guarantee. In Next.js App Router, utilities need to be flexible enough to handle both direct token passing and automatic header extraction (using `headers()` from `next/headers`).
**Prevention:** Always use established cryptographic methods (like HMAC) for security tokens. Use `crypto.timingSafeEqual` for comparing sensitive values. Ensure all state-changing API routes (POST, PUT, DELETE, PATCH) implement both origin validation and token verification.
