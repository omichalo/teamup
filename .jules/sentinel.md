## 2026-02-15 - Broken Object Level Authorization (BOLA) and Privilege Escalation in Firestore Rules
**Vulnerability:** The `availabilities` collection had overly permissive write rules (`allow write: if isAuthenticated()`), allowing any logged-in user to modify or delete any availability document. Additionally, the `users` collection allowed users to update their own documents without restricting sensitive fields like `role`, enabling self-privilege escalation.
**Learning:** Defaulting to simple `isAuthenticated()` checks for write operations is dangerous in multi-tenant or multi-user applications. Role-based access control (RBAC) should be enforced at the database level even if the frontend or middleware already has checks.
**Prevention:** Always use `request.resource.data.diff(resource.data).affectedKeys().hasAny([...])` to protect sensitive fields from client-side updates. For collection-wide write access, ensure it's restricted to authorized roles (e.g., `isCoach()` or `isAdmin()`) and use specific rules for owner-only updates if necessary.
